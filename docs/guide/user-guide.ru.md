# Sorta — Руководство пользователя (русский)

> Языки: [English](user-guide.en.md) · **Русский** · [日本語](user-guide.ja.md)

Sorta — инструмент командной строки и локального веб‑интерфейса, который
**индексирует большую коллекцию фото/видео** (протестировано на 60+ ГБ, рассчитано
на 300+ ГБ) и **раскладывает файлы в новую структуру папок** — по **городам/странам**,
по **людям** или по **событиям** — с полными гарантиями безопасности (по умолчанию
dry‑run, журнал перемещений и откат одной командой).

- **Локально по умолчанию.** Все ML‑модели (лица, определение сцен/текста) работают
  офлайн на вашей машине (рекомендуется GPU). Ничего не отправляется наружу, пока вы
  явно не включите онлайн‑провайдер в конфиге.
- **Оригиналы никогда не изменяются.** Сортировка *перемещает* или *копирует* файлы;
  EXIF не переписывается. С `--copy`/`--link` оригиналы остаются на месте.
- **Два способа работы:** пошаговый **веб‑интерфейс** (`sorta ui`) или **CLI**. Оба —
  обёртки над одним движком, выбирайте удобный.

## 2. Требования

| Компонент | Требование |
|---|---|
| ОС | Windows, Linux или macOS |
| Python | 3.11 – 3.14 (`requires-python >=3.11,<3.15`) |
| Менеджер окружения | [`uv`](https://docs.astral.sh/uv/) (рекомендуется) или `pip` |
| `exiftool` | **Обязателен** для метаданных HEIC/RAW/видео (даты, GPS, ориентация) — по сути, любое фото с современного телефона. Без него Sorta откатывается на Pillow, которая читает только JPEG/PNG/TIFF/WEBP и не читает видео. |
| Место на диске | Достаточно для новой структуры. `--copy` дублирует данные (×N). `--link` (жёсткие ссылки) почти не занимает места (тот же том, NTFS/ext4/APFS). Плюс SQLite‑индекс и (опционально) миниатюры — оба малы относительно самой коллекции. |

ML‑бэкенд Sorta (лица, CLIP/OCR для классификации мусора) ставится через один из
двух **взаимоисключающих установочных профилей** — выберите под своё железо:

| | Профиль CPU (`--extra cpu`) | Профиль GPU (`--extra gpu`) |
|---|---|---|
| Железо | Любая машина x86‑64, GPU не нужен | NVIDIA GPU + драйвер с поддержкой **CUDA 13** (проверено на Blackwell/RTX 5090) |
| Бэкенд | `onnxruntime` (CPU) + CPU‑сборки torch/torchvision | `onnxruntime-gpu` + рантайм CUDA 13/cuDNN 9 (pip‑wheels) + CUDA‑сборки torch/torchvision |
| Скорость лиц/CLIP | Работает корректно, но **медленно** — на большой коллекции `faces`/`junk`/`landmarks` займут часы. Хватает для раскладки по городам + дублей (лица/события всё равно opt‑in, см. §8), пригодно для небольших коллекций с включёнными лицами/событиями. | Быстро. Ориентир с нашей тестовой коллекции из 6298 фото, лица+события+junk включены: **≈ 45 мин** (быстрый CLIP‑ярус), **≈ 77 мин** с опциональным глубоким VLM‑ярусом (`naming.vlm_enabled` / `uv sync --extra vlm`). |
| RAM | Рекомендуется 8 ГБ+ (индексация/хэширование — самая RAM‑затратная часть, не зависит от профиля) | То же, плюс сколько зарезервирует драйвер GPU |
| VRAM | н/д | **~3 ГБ** для базы + лиц (замерено на RTX 5090: CLIP ViT‑L ≈2.0 ГБ + buffalo_l ≈0.6 ГБ) — комфортно на GPU с **≥4 ГБ**. Опциональный глубокий VLM‑ярус (Qwen2.5‑VL‑3B) добавляет ≈7 ГБ (оценка по 3B fp16, не замерено) → **≥8 ГБ** суммарно |

Тайминги и VRAM — наблюдения с нашего железа, не гарантия — на вашем результат
зависит от состава коллекции (видео‑превью, RAW, число лиц на фото — основные
факторы стоимости).

## 3. Установка

```bash
git clone https://github.com/shinKatana0/sorta.git
cd sorta

# Установить exiftool — ОБЯЗАТЕЛЕН для метаданных HEIC/RAW/видео:
#   Windows: winget install OliverBetz.ExifTool
#   Debian/Ubuntu: sudo apt install libimage-exiftool-perl
#   macOS: brew install exiftool

# Создать свой конфиг из шаблона
cp config.example.yaml config.yaml
```

ML‑бэкенду Sorta (лица, CLIP/OCR) нужен ровно один установленный профиль под
железо — `cpu` или `gpu`, взаимоисключающие (см. §2). Есть два поддерживаемых
способа его поставить, оба — **«поставил один раз → просто запускай `sorta`»** —
выбирайте по задаче:

### A) Глобальная установка через `uv tool install` (рекомендуется для обычной работы)

```bash
uv tool install ".[cpu]"        # без NVIDIA GPU
# или
uv tool install ".[gpu]"        # NVIDIA GPU + драйвер CUDA 13
```

Это резолвит ту же схему профилей/индексов из `pyproject.toml` (индексы
`pytorch-cu130` / `pytorch-cpu`), что и `uv sync`, и ставит команду `sorta` на
PATH — проверено: на профиле `gpu` реально даёт CUDA 13-сборку torch
(`torch.cuda.is_available()` → `True`). Дальше просто запускайте `sorta ui`,
`sorta index …` и т.д. из любого терминала, из любой папки — без `uv run`, без
активного venv.

- **Сменить профиль** (переехали на другое железо или выбрали не тот) —
  переустановите с `--force` и другим extra:
  `uv tool install --force ".[gpu]"` (или `".[cpu]"`).
- **Обновить после `git pull`** — `uv tool install --force ".[<профиль>]"`. Это
  ставит свежий снапшот текущего кода; это **не** editable-установка, так что
  для автоматического подхвата локальных правок нужен путь B ниже.
- Когда Sorta будет опубликована на PyPI, та же идея станет
  `uv tool install "sorta[gpu]"` (или `"sorta[cpu]"`) — без локального чекаута.

### B) Venv проекта через `uv sync` (для разработки над кодом)

```bash
uv sync --extra cpu --extra dev      # без NVIDIA GPU
# или
uv sync --extra gpu --extra dev      # NVIDIA GPU + драйвер CUDA 13

# Активировать один раз на сессию терминала:
.\.venv\Scripts\Activate.ps1         # Windows PowerShell
source .venv/bin/activate            # Linux/macOS/bash
```

С активным venv `sorta …` запускается прямо из вашего чекаута (editable-установка)
— правки кода видны сразу, переустановка не нужна.

> **Не запускайте `uv run sorta …` как повседневную команду.** `uv run <cmd>`
> перед каждым запуском пересинхронизирует окружение под базовый набор
> зависимостей из `pyproject.toml` — если не повторять `--extra <профиль>`
> на этой самой команде каждый раз, пересинк молча роняет GPU‑пакеты (torch
> откатывается на CPU‑сборку) при каждом запуске. Путь A (`uv tool install`) и
> путь B (активированный venv) полностью обходят эту проблему — в этом и весь
> смысл поставить один раз вместо запуска через `uv run`.

Всегда указывайте явный `--extra cpu` или `--extra gpu` в любом из путей — они
помечены взаимоисключающими в `pyproject.toml`, чтобы `uv` подобрал правильную
сборку torch/onnxruntime (GPU‑wheels тянут рантайм CUDA 13 обычными
pip‑пакетами, системный CUDA Toolkit не нужен). Профиль за вас никто не выберет.

`--extra dev` добавляет dev‑инструменты (ruff, mypy, pytest) — нужны, если будете
гонять `scripts/check.py` или тесты, для самого запуска `sorta` не обязательны.
Есть ещё опциональный `--extra vlm` для глубокого VLM‑яруса классификации
(`naming.vlm_enabled`); без него этот ярус аккуратно откатывается на быстрый
CLIP‑ярус.

## 4. Настройка

Sorta читает `config.yaml` (скопируйте из `config.example.yaml`). Два обязательных к
проверке параметра:

```yaml
sources:
  - "D:/Photos"          # папка(и) с фото/видео (сканируются рекурсивно)
database: "sorta.db"     # где хранится SQLite‑индекс
language: ru             # язык интерфейса/папок: ru | en | ja  (по умолчанию ru)
```

- **`sources`** — одна или несколько корневых папок для сканирования. Путь можно также
  передать в командной строке (`sorta index /путь/к/фото`) — он переопределяет конфиг.
- **`language`** — задаёт язык генерируемых **имён папок** (напр. `Россия/…` vs
  `Russia/…`) и **хрома веб‑интерфейса**. Поддерживаются: `ru`, `en`, `ja`.

> **Замечание:** `language` НЕ влияет на собственные консольные сообщения CLI
> (строки прогресса вроде `Готово: +13 новых, ...`) — это фиксированный текст,
> независимый от конфига. А вот имена папок и веб‑интерфейс локализуются полностью.
> Как выглядит реальный вывод CLI — см. примеры в §9, а если он превращается в
> `????` в терминале — см. §18.

Все параметры — в разделе [Справочник конфигурации](#17-справочник-конфигурации).

## 5. Ключевые понятия

**Индекс отделён от сортировки.** Сначала пайплайн наполняет SQLite‑**индекс**
(метаданные, гео, эмбеддинги лиц, кластеры, события, junk‑классификация). Сортировка —
это лишь *применение представления* индекса к файловой системе. Смена режима
(город ↔ люди ↔ события) **не** требует пересканирования.

**Dry‑run по умолчанию.** `sort` и `album` печатают план и ничего не пишут, пока вы не
добавите `--apply`. Всегда сначала смотрите план.

**Журнал и откат.** Каждое перемещение/копирование/линк записывается в журнал *до*
операции с ФС; `sorta undo` откатывает последний батч. Хэши (blake3) сверяются перед
перемещением; конфликт имён → суффикс `_1`, `_2` — существующий файл никогда не
перезаписывается.

**Три режима переноса.**
- **move** (по умолчанию для `sort`) — перемещает файл. Одна структура на диске.
- **copy** — дублирует файл; оригиналы не тронуты. Возможно несколько структур, ×N
  места.
- **link** (hardlink, по умолчанию для `album`) — второе имя тех же байтов; почти ноль
  лишнего места; фолбэк на copy при переносе между томами/ФС.

**Канон + альбомы.** Рекомендуемая модель: одна **каноническая** структура по городам,
плюс **альбомы по запросу** (конкретный человек / событие), собираемые в отдельные
именованные папки жёсткими ссылками.

## 6. Быстрый старт — веб‑интерфейс (рекомендуется)

Веб‑интерфейс — самый простой путь, из терминала нужен только запуск.

```bash
sorta ui                       # локальный сервер на http://127.0.0.1:8756
```

Далее в браузере:

1. Вкладка **Обработать** → укажите путь к папке с фото. Два чекбокса, по умолчанию
   **выключены**: **«Разбор по лицам»** и **«Разбор по событиям»** — самые долгие
   этапы пайплайна, opt‑in намеренно (см. §8). Оставьте их выключенными для быстрого
   прогона только по городам, либо включите нужное. Нажмите **Обработать**: пайплайн
   идёт в фоне с прогрессом по этапам (индекс → гео → landmarks → [лица] → [события]
   → junk → почти‑дубли — лица/события только если включены). Вкладку можно
   закрыть — обработка продолжится.
2. Вкладка **Города** → посмотрите предлагаемую структуру (`Страна/Город/Год/Район`).
   Видна всегда.
3. Вкладка **Дубли** → разберите группы почти‑дубликатов; рекомендованный кадр (★)
   выбран заранее. Поправьте, где не согласны, и один раз нажмите **Сохранить весь
   выбор**. Видна всегда.
4. Вкладка **Люди** → появляется только когда есть кластеры лиц (вы хоть раз включали
   «Разбор по лицам» или запускали `sorta faces`). Назовите кластеры и слейте дубли
   одного человека.
5. Вкладка **События** → появляется только когда есть события (вы хоть раз включали
   «Разбор по событиям» или запускали `sorta events`). Переименуйте события; соберите
   любого человека/событие в папку кнопкой **Собрать в папку**.
6. Вкладка **Перемещения** → после применения сортировки/альбома видно, что куда
   легло. Видна всегда.

На вкладке **Обработать** есть ещё два чекбокса, помимо лиц/событий, — оба
отражают `config.yaml` и работают как полный override только для этого прогона
(отметил = принудительно вкл, снял = принудительно выкл) — UI-эквивалент CLI-флагов
`--deep`/`--no-deep` и `--geo online`/`--geo offline` (§8):

- **«Глубокий анализ (VLM)»** — использовать глубокий VLM-ярус вместо быстрого
  CLIP-яруса для классификации junk/документов. Реально включается, только если
  он *одновременно* запрошен (этот чекбокс, `--deep`, либо
  `naming.vlm_enabled: true` в конфиге) *и* установлен (extra `vlm`, например
  `uv tool install ".[gpu,vlm]"` или `uv sync --extra gpu --extra vlm --extra
  dev`) — без этого extra он молча откатывается на быстрый CLIP-ярус, о чём и
  говорит подсказка под чекбоксом.
- **«Онлайн-гео (точнее заграница)»** — использовать онлайн-геокодинг Nominatim
  вместо офлайн-данных GeoNames для этого прогона; отправляет только
  GPS-координаты, никогда не фото (см. §15).

Если после прогона «Люди»/«События» не появились — это ожидаемо: значит лица/события
не были включены для этого прогона, а не что-то сломалось; перезапустите с
включённым чекбоксом (или `sorta faces`/`sorta events`) — вкладка появится.

Сервер слушает только `127.0.0.1` (недоступен из сети). Остановка — `Ctrl+C`.

## 7. Быстрый старт — CLI

Примеры ниже предполагают, что `sorta` уже на PATH — через `uv tool install` или
активированный venv (§3). Не добавляйте перед ними `uv run` — почему, см.
предупреждение в §3.

```bash
# 1) Индексация папки (метаданные, хэши, точные дубли)
sorta index /путь/к/фото

# 2) Базовый пайплайн (гео, landmarks, junk) + хэши почти‑дублей — без лиц/событий
sorta run
sorta phash

# 2b) ...или включить ещё и лица/события (самые долгие этапы, см. §8):
sorta run --faces --events

# 3) Предпросмотр сортировки по городам (dry-run — CSV + HTML план, ничего не двигает)
sorta sort --by city --dest /путь/к/разложенному

# 4) Применить (copy неразрушающий; без --copy будет MOVE)
sorta sort --by city --dest /путь/к/разложенному --copy --apply

# Откат последнего батча при необходимости
sorta undo
```

### Пример от начала до конца

Ниже — **реальный вывод команд**, снятый на небольшой синтетической тестовой
коллекции (13 сгенерированных JPEG со встроенными EXIF/GPS: двухдневная поездка
«Париж», день «Токио», который слишком мал, чтобы стать событием, точный дубль,
почти‑дубль, скриншот и два фото‑заглушки «лиц», нужные только чтобы прогнать
пайплайн — не настоящие фотографии кого-либо). Показано, чтобы вы точно знали,
чего ждать; полный разбор каждого режима — дальше в §9–§13.

```
$ sorta index -c config.yaml
Готово: +13 новых, ~0 обновлено, 0 пропущено, 0 ошибок, 1 дубликатов помечено

$ sorta geo -c config.yaml
Готово: 12 файлов — exact_gps 10, session_inferred 1, unknown 1

$ sorta faces -c config.yaml
Детекция: 12 файлов, 0 лиц, 12 без лиц, 0 ошибок
Кластеры: 0 (лиц в кластерах: 0, шум: 0, имён сохранено: 0)

$ sorta events -c config.yaml
События: 1 авто (7 файлов, имён сохранено: 0), 0 ручных (0 файлов)

$ sorta junk -c config.yaml
Классификация: 12/12 обработано (photo: 11, screenshot: 1)

$ sorta phash -c config.yaml
pHash посчитан для 13 фото. Отчёт: sorta dupes --near

$ sorta stats -c config.yaml
Файлов в индексе: 13 (+0 с ошибками)
  с GPS:            11 (84%)
  дата из exif     : 13 (100%)
  дата из filename : 0 (0%)
  дата из mtime    : 0 (0%)
  дубликатов:       1
Гео (places): 12
  exact_gps       : 10 (83%)
  unknown         : 1 (8%)
  session_inferred: 1 (8%)
```

Несколько деталей, которые стоит заметить (реальные, не подогнаны для красоты):

- `index` нашёл **13** файлов, и `stats` позже тоже говорит 13 — файл‑точный‑дубль
  *индексируется* (с проставленным `dup_of`), просто у него нет своей строки
  места/события/junk, поэтому `geo`/`junk` отчитываются о **12**.
- `faces` реально нашёл **0 лиц** на двух заглушках — настоящий детектор
  фотографических лиц не срабатывает на плоской векторной графике, поэтому мы
  намеренно не стали выдумывать здесь пример «найдено 2 лица, названы Аней». Как
  выглядит работа с людьми на реальных фото — см. §11.
- `events` построил **1** событие из **7** файлов (поездка в Париж); 4 файла из
  Токио остались ниже `events.min_event_size` (5) и при сортировке по событиям
  (§9) уходят в резервную корзину `no_event` — реальная, полезная демонстрация
  этого порога, а не баг.
- `landmarks` не показан выше, потому что на этих данных ему нечего было делать
  (ни один файл без GPS не оказался достаточно близко к настоящей
  достопримечательности из bundled‑каталога) — см. §9, как этот этап работает,
  когда срабатывает.

## 8. Пайплайн обработки

`sorta run` (или кнопка **Обработать** в UI) выполняет этапы по порядку. Каждый —
отдельная команда, и все **инкрементальны** (повторный запуск обрабатывает только
новые/изменённые файлы):

| Этап | Команда | По умолчанию? | Что делает |
|---|---|---|---|
| Индекс | `sorta index [dir]` | всегда | Сканирует файлы, читает EXIF/даты, считает blake3, помечает точные дубли. |
| Гео | `sorta geo` | всегда | Определяет место по GPS; для файлов без GPS наследует место от соседей по времени (офлайн GeoNames или онлайн Nominatim, если включён). |
| Landmarks | `sorta landmarks` | всегда | Визуальное определение места для сцен без GPS, консервативный порог — заполняет город, например, для фото достопримечательности в помещении без GPS. |
| Лица | `sorta faces` | **opt‑in** (`--faces`) | Детектирует лица (insightface), считает эмбеддинги, кластеризует людей (HDBSCAN). Самый долгий этап; пропускается, если явно не запрошен. |
| События | `sorta events` | **opt‑in** (`--events`) | Группирует фото в события по разрывам времени + городу; именует по дате + городу. Независим от лиц — включайте любое, оба или ни одного. |
| Junk | `sorta junk` | всегда | Классифицирует фото: `photo` / `screenshot` / `meme` / `document` (эвристики + CLIP + плотность текста). |
| Хэши почти‑дублей | `sorta phash` | всегда (UI); в CLI отдельная команда (`sorta run` её не вызывает — гоняйте `sorta phash` сами) | Перцептивные хэши для почти‑дубликатов. |

**Флаги `sorta run`** (все опциональны, все — override только на ЭТОТ прогон,
в `config.yaml` ничего не пишется):

```
--faces / --no-faces       Детекция+кластеризация лиц в этом прогоне (по умолчанию: выкл)
--events / --no-events     Построение событий в этом прогоне (по умолчанию: выкл)
--deep / --no-deep         Глубокий VLM-ярус классификации junk в этом прогоне
                            (нужен `uv sync --extra vlm`; без него аккуратный откат
                            на быстрый CLIP-ярус). По умолчанию — из config.yaml
                            (naming.vlm_enabled).
--geo offline|online       Провайдер геокодинга для этого прогона. `online` точнее
                            за границей, но отправляет GPS-координаты (не фото)
                            в Nominatim. По умолчанию — из config.yaml (geo.provider).
--by city|person|event     Дополнительно напечатать dry-run план сортировки в конце (§9)
--dest DIR                 Куда этот план (без него — in-place)
```

**Базовый прогон** (`sorta run` без флагов) намеренно быстрый путь: раскладка по
городам и поиск дублей, больше ничего. Включайте `--faces`/`--events`, когда
действительно нужна сортировка/альбомы по людям или событиям — отдельный последующий
запуск `sorta faces`/`sorta events` работает точно так же и в любом случае полностью
инкрементален. Покрытие в любой момент — `sorta stats`.

## 9. Сортировка: города, люди, события

```bash
sorta sort --by city   --dest <dir> [--apply] [--copy|--move] [--where …] [--dedupe]
sorta sort --by person --dest <dir> [--apply] …
sorta sort --by event  --dest <dir> [--apply] …
```

- **`--by city`** → `Страна/Город/Год/Район/…` (локализованные имена).
- **`--by person`** → папка на каждого **названного** человека (сначала назовите
  кластеры — см. §11).
- **`--by event`** → `Год/ИмяСобытия/…`.
- **`--dest`** — целевой корень. Без него сортировка **in‑place** (реструктурирует
  саму исходную папку — dry‑run, журнал и undo всё равно действуют).
- **`--copy` / `--move`** — копировать (оригиналы сохраняются) или перемещать (по
  умолчанию).
- **`--where`** — фильтр плана, повторяемый: `--where "country=DE" --where "year>=2020"`.
- **`--dedupe`** — увести худшие почти‑дубли в папку `_Duplicates`.
- **`--exclude <path>`** — пропустить уже разобранную подпапку.

Файлы, не подходящие под режим, уходят в review‑папки: `_Неразобрано/` (нет места /
даты / junk), `_Документы/` (см. §14).

Без `--apply` — **dry‑run**: CSV + просматриваемый HTML‑план в `report_output/` рядом с базой, и
**ничего не перемещается**.

### Пример — `--by city`

Продолжаем синтетическую коллекцию из §7 (index/geo/junk уже прогнаны):

```
$ sorta sort --by city --dest sorted -c config.yaml
sort --by city (dry-run): 12 файлов -> 4 каталогов; план: …\report_output\sort_plan_city_20260721_113247.csv, …\report_output\sort_plan_city_20260721_113247.html
```

План CSV (по строке на файл — `target` относительно `--dest`) — сокращён до нужных
колонок:

| path | country | city | target | reason |
|---|---|---|---|---|

| `paris_01.jpg` | FR | Paris | `Франция/Париж/2023/paris_01.jpg` | city |
| `paris_02.jpg` | FR | Paris | `Франция/Париж/2023/paris_02.jpg` | city |
| `paris_02_edited.jpg` | FR | Paris | `Франция/Париж/2023/paris_02_edited.jpg` | city |

Применяем (`--copy`, чтобы оригиналы остались на месте — без флага будет перемещение)

$ find sorted_apply -type f

sorted_apply/Франция/Париж/2023/paris_02.jpg
sorted_apply/Франция/Париж/2023/paris_02_edited.jpg
sorted_apply/Франция/Париж/2023/paris_03.jpg
sorted_apply/Франция/Париж/2023/paris_04.jpg
sorted_apply/Франция/Париж/2023/paris_05_nogps.jpg

sorted_apply/Япония/Токио/2023/tokyo_02.jpg

(ничего — undo убрал все копии)

На `language: ja` та же сортировка даёт `フランス/パリ/2023/…` и, что интересно,
`日本/東京都/2023/桜丘町/…` — подпапку **района** (Сакурагаока), которой нет для

локализованные имена районов, которых нет для `en`/`ru`, и
`naming.drop_unlocalized_district` (по умолчанию включён) скрывает сегмент района
для языка, на который его нельзя локализовать, вместо показа сырого
транслитерированного кода.

**Фильтрация через `--where`:**

```
$ sorta sort --by city --dest sorted_fr --where "country=FR" -c config.yaml

```

В план попадают только 7 файлов, разрешённых во Францию; остальное вообще не входит
в план (не уходит в `_Неразобрано`).

### Пример — `--by event`

```
$ sorta sort --by event --dest sorted_event -c config.yaml
sort --by event (dry-run): 12 файлов -> 3 каталогов; план: …
```

| path | event | target |
|---|---|---|
| `paris_01.jpg` … `person_a_1.jpg` (7 файлов) | `2023-06-10..06-11 Paris` | `2023/2023-06-10..06-11 Paris/<имя>.jpg` |
| `tokyo_01.jpg`, `tokyo_02.jpg`, `tokyo_03.jpg`, `person_a_2.jpg` (4 файла) | *(нет — ниже `events.min_event_size`)* | `2023/11/<имя>.jpg` — резерв `no_event`, группировка по году/месяцу вместо этого |
| `shot_01.jpg` | | `_Неразобрано/мусор/screenshot/shot_01.jpg` — junk побеждает в любом режиме |

Это тот же порог `min_event_size` из §7/§12 в деле: у токийского дня были реальный
GPS, реальные метки времени, реальное место — всё, кроме *достаточного числа файлов*,
чтобы стать самостоятельным именованным событием.

### Пример — `--by person`

Режиму person нужны сначала **названные кластеры лиц** (§11), а для них — реальные
лица, которые `sorta faces` находит на настоящих фотографиях — то, что наши
синтетические заглушки честно продемонстрировать не могут (см. заметку в §7 и
оговорку в §11). После того как вы назвали пару кластеров на реальной коллекции,
это выглядит так:

```bash
sorta sort --by person --dest /путь/к/разложенному --apply
```

даёт `<dest>/<ИмяЧеловека>/<файл>.jpg` для каждого фото, где этот человек —
(или главный, см. `sort.multi_person` в §17) названное лицо — всему остальному без
названного человека всё равно нужно куда-то деться, поэтому такие фото уходят в
`_Неразобрано/`. Маршрутизация junk/скриншотов и `--where`/`--copy`/`--move`/`--apply`
работают точно так же, как в примерах city/event выше.

## 10. Дубликаты

- **Точные дубли** (идентичные байты) находятся при `index`; сортируется только
  канонический экземпляр, остальные остаются на месте.
- **Почти‑дубли** (визуально похожие, разного размера/имени) — через перцептивные хэши
  (`sorta phash`, затем `sorta dupes --near` или вкладка **Дубли** в UI).

Во вкладке **Дубли**: у каждой группы рекомендованный кадр (★). Поправьте radio, где не
согласны, отметьте *«не удалять эту группу»* для пропуска, затем один раз нажмите
**Сохранить весь выбор** (без покадрового клика). При следующей сортировке/копии
не‑keeper'ы уйдут в папку `_удалить` (восстановимо) — либо кнопкой **Удалить** на кадре /
**Удалить дубли** отправьте их сразу в корзину ОС.

Реальный вывод на синтетической коллекции из §7 (после `sorta phash`):

```
$ sorta dupes -c config.yaml
paris_01_copy.jpg
  -> дубликат paris_01.jpg

Всего: 1

$ sorta dupes --near -c config.yaml
Группа из 2 похожих:
  paris_02.jpg  (7424 байт)
  paris_02_edited.jpg  (5908 байт)
Группа из 2 похожих:
  person_a_1.jpg  (14742 байт)
  person_a_2.jpg  (14742 байт)

Групп: 2 (порог Хэмминга: 5)
```

`paris_02_edited.jpg` — реально пересжатая/уменьшенная копия `paris_02.jpg`, ровно
случай «то же фото, отредактировано или переэкспортировано», для которого и нужен
перцептивный хэш. Вторая группа — ложное, но поучительное срабатывание: два наших
фото‑заглушки байт‑в‑байт идентичны (сгенерированы одной процедурой), поэтому pHash
корректно называет их почти‑дублями, хотя `sorta faces` считает их несвязанными
файлами (лиц не найдено ни на одном). На реальной коллекции два разных фото одного
человека обычно НЕ будут почти‑дублями — pHash сравнивает похожесть всего кадра, а
не личность.

## 11. Люди и кластеры лиц

Детекция лиц даёт **кластеры** (группы одного лица). Чтобы person‑сортировка была
осмысленной, назовите кластеры:

- **UI → вкладка Люди:** у кластера видны образцы лиц; введите имя и нажмите
  **Назвать**; выберите два кластера и **Слить**, если это один человек.
- **CLI:** `sorta faces label <cluster_id> "Мама"`, `sorta faces merge <src> <dst>`,
  `sorta faces sheet <cluster_id> out.html` (контактный лист для опознания).

После именования `sorta sort --by person` (или person‑**альбом**, §13) использует имена.

`sorta faces` нужен флаг `--faces` у `sorta run` / чекбокс «Разбор по лицам» в UI
(§8) — на базовом пайплайне он не запускается. Реальный вывод на синтетической
коллекции из §7:

```
$ sorta faces -c config.yaml
Детекция: 12 файлов, 0 лиц, 12 без лиц, 0 ошибок
Кластеры: 0 (лиц в кластерах: 0, шум: 0, имён сохранено: 0)
```

Действительно ноль — buffalo_l обучена на настоящих фотографиях и корректно не
срабатывает на наших синтетических заглушках (плоские векторные фигуры, не реальная
текстура лица). Это ожидаемо, не баг Sorta и не баг этого руководства: направьте
`sorta faces` на настоящую фотоколлекцию — и он найдёт реальные лица. Когда найдёт
(реальный, не этот синтетический прогон напечатал бы что-то вроде `Детекция: 340
файлов, 512 лиц, 8 без лиц, 0 ошибок` / `Кластеры: 6 (лиц в кластерах: 480, шум: 32,
имён сохранено: 0)`), именование и сортировка — те же команды выше:
`sorta faces label 3 "Мама"` называет кластер `3`, затем
`sorta sort --by person --dest … --apply` раскладывает фото этого человека в
`<dest>/Мама/`.

## 12. События

События группируют фото по разрывам времени и городу. `sorta events` (пере)строит их:

- Мелкие группы (меньше `events.min_event_size`) событиями не становятся.
- Сессии одного города в пределах `events.trip_merge_gap_hours` сливаются в поездку.
- Имя = диапазон дат + локализованный город (напр. `2023-11-29..12-02 Сочи`).

Ручное управление:
- `sorta events add "Конференция" 2025-05-21 2025-05-23` — ручное событие на диапазон
  дат (переживает пересчёт).
- `sorta events rename <event_id> "IEEE конференция Токио"` — ручное имя.

`sorta events` нужен флаг `--events` у `sorta run` / чекбокс «Разбор по событиям» в
UI (§8). Реальный вывод на синтетической коллекции из §7 (7 парижских файлов
проходят порог `min_event_size` по умолчанию — 5; токийский день из 4 файлов — нет):

```
$ sorta events -c config.yaml
События: 1 авто (7 файлов, имён сохранено: 0), 0 ручных (0 файлов)
```

## 13. Альбомы

**Альбом** собирает конкретный срез — одного человека (опционально с фильтром) или одно
событие — в отдельную именованную папку, не трогая каноническую структуру по городам.

```bash
# Все фото «Мама», жёсткими ссылками (по умолчанию), сначала превью, затем apply:
sorta album person "Мама" --dest /путь/к/альбомам
sorta album person "Мама" --dest /путь/к/альбомам --apply

# «Мама», но только в Барселоне:
sorta album person "Мама" --where "city=Barcelona" --dest /путь/к/альбомам --apply

# Конкретное событие с своим именем папки, копиями:
sorta album event "2025-05-21..05-23 Tokyo" --dest /путь/к/альбомам \
      --name "IEEE конференция Токио" --copy --apply
```

- Режим по умолчанию — **link** (hardlink, ~0 лишнего места; фото может быть в
  нескольких альбомах *и* в структуре по городам).
- **`--copy`** — независимые копии; **`--move`** *изымает файлы из общего пула*
  (печатает предупреждение). Фото с **2+ названными людьми** нельзя move в один альбом
  (неоднозначно) — такие блокируются; используйте link/copy.
- В UI — кнопка **Собрать в папку** на карточках Люди/События.

Реальный вывод — собираем парижское событие из §12 в альбом, копиями:

```
$ sorta album event "2023-06-10..06-11 Paris" --dest albums --copy --apply -c config.yaml
album event '2023-06-10..06-11 Paris' --apply [copy]: 7 файлов -> …\albums\2023-06-10..06-11 Paris
Альбом «2023-06-10..06-11 Paris»: выгружено 7, ошибок 0. Откат: sorta undo

$ find albums -type f
albums/2023-06-10..06-11 Paris/paris_01.jpg
albums/2023-06-10..06-11 Paris/paris_02.jpg
albums/2023-06-10..06-11 Paris/paris_02_edited.jpg
albums/2023-06-10..06-11 Paris/paris_03.jpg
albums/2023-06-10..06-11 Paris/paris_04.jpg
albums/2023-06-10..06-11 Paris/paris_05_nogps.jpg
albums/2023-06-10..06-11 Paris/person_a_1.jpg
```

Поскольку передан `--copy`, это независимые файлы — `sorta undo` здесь удаляет
только копии в альбоме, оригиналы не трогает (см. §15).

## 14. Мусор, скриншоты и документы

`sorta junk` классифицирует каждое фото, чтобы сортировка убрала не‑воспоминания из
папок городов/людей/событий:

- **`screenshot`**, **`meme`** → `_Неразобрано/junk/…`. Файлы в папке `Screenshots/`
  распознаются и по имени папки.
- **`document`** (паспорта, чеки, бланки, медицинские бумаги…) → `_Документы/` —
  **review‑папка**, *не* мусор. Детекция сочетает CLIP с сигналом **плотности текста**
  (документы плотно покрыты текстом; пляжи и товарные кадры — нет).

`_Документы/` намеренно **over‑собирает** (реальное фото оттуда легко вытащить; реальный
документ, утёкший в память по городам — хуже). Разбирайте вручную. Замечание:
автоматическое отделение фото «товаров на продажу» от документов — известное
ограничение, они могут смешиваться в `_Документы/`.

> **Приватность:** документы могут содержать персональные данные. Sorta обрабатывает их
> **локально** и никогда не отправляет наружу (если вы не включили онлайн‑провайдер).
> См. §15.

Реальный вывод на синтетической коллекции из §7 — единственное фото, сохранённое в
`Screenshots/`, распознаётся эвристикой по имени папки, остальные классифицируются
как обычные фото:

```
$ sorta junk -c config.yaml
Классификация: 12/12 обработано (photo: 11, screenshot: 1)
```

## 15. Безопасность, откат и приватность

- **Dry‑run по умолчанию** — ничего не двигается до `--apply`.
- **Журнал перемещений** — каждая операция записывается *до* выполнения.
- **Откат** — `sorta undo` откатывает последний батч (`--batch <id>` для конкретного).
  Для copy/link‑батчей undo удаляет копии/ссылки, а не оригиналы.
- **Сверка хэшей, без перезаписи** — blake3 проверяется перед перемещением; конфликт
  имён → `_1`, `_2`.
- **Оригиналы не тронуты с copy/link.** С move файлы переезжают, но содержимое и EXIF
  неизменны.
- **Локально по умолчанию.** Модели лиц/сцен/текста работают на вашей машине.
  Онлайн‑провайдеры — **opt‑in** в `config.yaml`: `geo.provider: online` (Nominatim)
  отправляет только GPS‑координаты, никогда не изображения; `naming.provider: claude`
  отправляет несколько фото‑образцов события в Claude API (единственная функция,
  реально покидающая машину с фото) — что именно отправляет каждый провайдер, см.
  [SECURITY.md](../../SECURITY.md). Для максимальной приватности держите их
  выключенными.
- Веб‑интерфейс слушает только `127.0.0.1`.

## 16. Полный справочник команд

```
sorta index [DIR]                 Сканировать sources (или DIR) → метаданные, хэши, точные дубли
sorta run [--src DIR] [--faces] [--events] [--deep/--no-deep] [--geo offline|online]
          [--by city|person|event] [--dest DIR]
                                  Базовый пайплайн (index→geo→landmarks→junk); --src
                                  переопределяет config sources на этот прогон; --faces/
                                  --events включают долгие этапы (по умолчанию выкл,
                                  независимо друг от друга); --deep/--geo override
                                  config.yaml только на этот прогон; с --by в конце
                                  печатает dry-run план
sorta geo                         Определить места (GPS + наследование по сессиям)
sorta landmarks                   Визуальное место для сцен без GPS (консервативно)
sorta faces                       Детекция лиц + кластеризация людей
sorta faces label <cluster> <name>    Назвать кластер
sorta faces merge <src> <dst>          Слить два кластера (один человек)
sorta faces sheet <cluster> <out.html> Контактный лист для опознания
sorta events                      (Пере)построить события
sorta events add <name> <from> <to>    Ручное событие на диапазон дат
sorta events rename <id> <name>        Ручное имя события
sorta junk                        Классификация photo/screenshot/meme/document
sorta phash                       Перцептивные хэши (для почти‑дублей)
sorta stats                       Покрытие индекса (GPS, источники дат, дубли)
sorta dupes [--near]              Список точных / почти‑дублей
sorta sort --by MODE [--dest DIR] [--apply] [--copy|--move]
           [--where …] [--dedupe] [--delete-worse-dupes] [--exclude PATH] [--thumbnails]
                                  Спланировать/применить сортировку (без --apply — dry-run)
sorta album person|event <selector> --dest DIR [--copy|--move] [--where …] [--name N] [--apply]
                                  Собрать срез в именованную папку (по умолчанию hardlink)
sorta undo [--batch ID]           Откатить последний (или указанный) батч
sorta reset [--yes]               Стереть индекс (БД) и начать с нуля — фото и уже
                                  разложенные папки НЕ трогает (имена людей/событий
                                  и решения по дублям при этом пропадут)
sorta ui [--port 8756]            Локальный веб‑апп (Обработать / Города / Дубли / Люди / События / Перемещения)
```

Каждая команда принимает `-c/--config <path>` (по умолчанию `config.yaml`).

  trip_merge_gap_hours: 48     # сессии одного города в пределах — в одну поездку

## 18. Решение проблем

- **`uv sync` без extra оставляет `sorta faces`/`sorta junk` сломанными или
  несогласованными** — ожидаемо. Всегда ставьте с явным профилем:
  `uv sync --extra cpu --extra dev` или `uv sync --extra gpu --extra dev` (§2/§3).
  `cpu`/`gpu` взаимоисключающие; смена железа позже — просто повторный `uv sync` с
  другим extra.
- **`No module named ruff` / нет dev‑инструментов** — добавьте `--extra dev` к
  вашему `uv sync` (это отдельно от профиля cpu/gpu, см. выше).
- **Нет дат/превью/метаданных видео для HEIC/RAW** — установите `exiftool` (см. §3);
  он обязателен для этих форматов, Pillow покрывает только JPEG/PNG/TIFF/WEBP.
- **Лица/CLIP очень медленно на GPU‑профиле** — убедитесь, что реально стоял
  `uv sync --extra gpu` (не `cpu`) и что драйвер поддерживает CUDA 13; `sorta faces`/
  `sorta junk` печатают, какой execution provider onnxruntime выбрал
  (`CUDAExecutionProvider` vs `CPUExecutionProvider`) в начале вывода.
- **Классификация/лица медленные, хотя ставили `--extra gpu`** — скорее всего,
  вы запускаете `sorta` через голый `uv run sorta …`. `uv run` перед каждым
  запуском пересинхронизирует окружение под базовые зависимости из
  `pyproject.toml`, что откатывает GPU‑сборку torch на CPU, если не повторять
  `--extra gpu` на этой самой команде каждый раз (см. §3). Запускайте бинарь от
  `uv tool install` (`uv tool install ".[gpu]"`, дальше просто `sorta …`) либо
  активированный venv — ни то, ни другое не пересинкается на каждый вызов.
  Проверка GPU: `python -c "import torch; print(torch.cuda.is_available())"`
  должно напечатать `True`. Сменить профиль железа = переустановить с другим
  extra — тот же шаг, что описан в §3.
- **Намеренно форсировать CPU на GPU‑профиле** (например, для отладки, или GPU занят
  чем-то другим) — задайте `CUDA_VISIBLE_DEVICES=` (пусто) для команды; и torch, и
  onnxruntime это учитывают и откатываются на CPU:
  ```bash
  CUDA_VISIBLE_DEVICES= sorta faces          # bash/macOS/Linux
  ```
  ```powershell
  $env:CUDA_VISIBLE_DEVICES=''; sorta faces  # PowerShell
  ```
- **`buffalo_l` перекачивается заново каждый раз** — кэш модели
  (`~/.insightface/models/buffalo_l`) удалён или недоступен для записи; убедитесь,
  что этот путь (или симлинк/junction туда, где реально лежит модель) сохраняется
  между запусками.
- **`database is locked`** — другой процесс Sorta пишет (напр. прогон пайплайна).
  Дождитесь завершения; не запускайте двух писателей одновременно.
- **Папка с не‑ASCII именем (напр. кириллица) как будто пропущена OCR** — исправлено:
  изображения декодируются через Unicode‑безопасный путь; обновитесь до последней
  версии.
- **Консольные сообщения CLI печатаются по-русски даже при `language: en`/`ja`** —
  ожидаемо, не баг: `language` управляет именами папок и веб‑интерфейсом, а не
  собственным текстом прогресса CLI (см. §4 и пример в §7). Если он ещё и
  превращается в `????` в терминале — это отдельная, чисто косметическая проблема
  кодировки (следующий пункт).
- **Кириллица/японский искажаются в Windows‑консоли** — косметика; файлы и веб‑интерфейс
  не затронуты. Используйте веб‑интерфейс, UTF‑8‑терминал, либо задайте
  `PYTHONUTF8=1` перед запуском `sorta`.
- **`sorta landmarks` (или другая команда) падает с ошибкой относительного пути
  вида «`data/landmarks.yaml` не найден»** — этот путь (`naming.landmarks_file` в
  `config.yaml`) резолвится относительно **текущей директории**, а не репозитория.
  Либо запускайте `sorta` из корня репозитория, либо задайте абсолютный путь для
  `naming.landmarks_file` в своём `config.yaml`.
